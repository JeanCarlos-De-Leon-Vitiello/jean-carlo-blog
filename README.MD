Weekly Nerd: Peter-Paul Koch
Wat hij vertelde
Tijdens deze Weekly Nerd vertelde Peter-Paul Koch over de geschiedenis van browsers en hoe browsers websites renderen. Hij legde uit welke onderdelen een browser heeft, zoals HTML, CSS, de DOM en de rendering engine.
Ook vertelde hij over render blocking. CSS en JavaScript kunnen ervoor zorgen dat een pagina later geladen wordt. Daarom zijn technieken zoals async en defer belangrijk voor performance.
Daarnaast ging hij in op backward compatibility. Browsers blijven oude code ondersteunen zodat oude websites nog steeds blijven werken.

Rendering engine
Verantwoordelijk voor het verwerken van HTML en CSS en het renderen van de pagina. Browsers zoals Chrome gebruiken hiervoor Blink.
AOM (Accessibility Object Model)
Naast de DOM bouwt de browser ook een AOM op. Deze wordt gebruikt door screenreaders en andere accessibility tools.
Core vs DOM
JavaScript Core gaat over de programmeertaal zelf, zoals variabelen en functies. De DOM zorgt ervoor dat JavaScript HTML-elementen kan aanpassen.
Render blocking
CSS blokkeert standaard rendering. JavaScript ook, tenzij je async of defer gebruikt.
defer: uitvoeren nadat de DOM geladen is.
async: direct uitvoeren zodra het bestand geladen is.
Backward compatibility
Browsers blijven oude technieken ondersteunen zodat oude websites nog steeds blijven werken.
Browser geschiedenis
Browsers zoals Mosaic, Netscape, Internet Explorer en Chrome hebben veel invloed gehad op hoe het web vandaag werkt.

Wat ik meeneem
Ik neem vooral mee dat browsers veel complexer zijn dan ik eerst dacht. Er gebeurt veel meer achter de schermen voordat een website zichtbaar wordt.
Daarnaast wil ik mij meer verdiepen in rendering engines zoals Blink en WebKit, omdat deze invloed hebben op hoe websites werken in verschillende browsers.



Weekly Nerd: Nils Binder
Wat hij vertelde
Tijdens deze Weekly Nerd vertelde Nils Binder over zijn bedrijf 9elements en liet hij verschillende projecten zien. Hij vertelde hoe design en development binnen hun team samenwerken om creatieve websites te bouwen.
Wat ik interessant vond was dat een Figma design vaak nog erg verandert tijdens development. Het eindproduct ziet er dus niet altijd precies hetzelfde uit als het eerste ontwerp.
Ook vertelde hij dat het web tegenwoordig meer mogelijkheden heeft dan Figma. Vroeger konden designs uit Photoshop vaak niet volledig op het web gemaakt worden, maar nu is dat juist andersom.
Daarnaast liet hij moderne CSS technieken zien zoals subgrid, view transitions en steps animations.

Belangrijke inzichten
Figma lijkt op CSS
Figma gebruikt principes die erg lijken op CSS, zoals auto layout dat vergelijkbaar werkt met flexbox.
Meer mogelijk op het web
Tegenwoordig kan het web meer dan design tools zoals Figma. Niet alles wat in CSS mogelijk is kan in Figma gemaakt worden.
Steps animations
Met steps() kan je een animatie maken met meerdere afbeeldingen zodat het lijkt op een flipbook of 3D animatie.
Subgrid
Met subgrid kunnen child elementen dezelfde grid structuur gebruiken als de parent grid.
View transitions
Met view transitions kan de browser automatisch animaties maken tussen pagina’s zonder JavaScript frameworks.
First-line selector
Met ::first-line kan je alleen de eerste regel van een tekst apart stylen.

Wat ik meeneem
Ik neem vooral mee dat webdesign veel verder is dan alleen Figma namaken. CSS biedt tegenwoordig veel creatieve mogelijkheden die ik nog niet kende.
Daarnaast wil ik meer experimenteren met moderne CSS features zoals subgrid en view transitions om creatievere websites te bouwen.




Weekly Nerd: Robert Broersma – FF formulieren maken
Wat hij vertelde
Tijdens deze Weekly Nerd vertelde Robert Broersma over het ontwerpen van goede formulieren. Hij legde uit dat formulieren niet alleen bestaan uit invoervelden, maar dat gebruiksvriendelijkheid en toegankelijkheid een grote rol spelen.
Hij vertelde bijvoorbeeld dat je alleen informatie moet vragen die echt nodig is. Ook is het belangrijk om autocomplete te gebruiken, duidelijke foutmeldingen te tonen en invoervelden logisch te ordenen.
Daarnaast ging hij in op accessibility, zoals voldoende kleurcontrast, duidelijke labels en grote klikvlakken voor gebruikers.

Belangrijke inzichten
Alleen noodzakelijke informatie
Vraag alleen gegevens die echt nodig zijn. Volgens de wet mag je geen onnodige informatie verzamelen.
Niet verplichte velden aangeven
In plaats van alle verplichte velden te markeren, kan je beter alleen de optionele velden aangeven.
Autocomplete gebruiken
Autocomplete maakt formulieren sneller en makkelijker om in te vullen.
Duidelijke foutmeldingen
Na het verzenden moet duidelijk zichtbaar zijn welke velden fouten bevatten.
Grote klikvlakken
Formulierelementen moeten groot genoeg zijn. Ongeveer 44x44px werkt gebruiksvriendelijk.
Zichtbare labels
Elk invoerveld moet een duidelijke label hebben die boven het veld staat.
Logische flow
Velden moeten in een logische volgorde staan zodat het formulier natuurlijk aanvoelt.
Opslaan en later doorgaan
Een formulier tussendoor kunnen opslaan zorgt voor een betere gebruikerservaring.

Wat ik meeneem
Ik neem vooral mee dat kleine details een formulier veel gebruiksvriendelijker maken. Dingen zoals autocomplete, duidelijke labels en goede foutmeldingen maken een groot verschil voor de gebruiker.
Daarnaast wil ik bij toekomstige formulieren meer rekening houden met toegankelijkheid en duidelijkheid.



Weekly Nerd: Hacker Filosofie
Wat zij vertelde
Tijdens deze Weekly Nerd ging het over hacker filosofie. Hierbij draait het niet alleen om hacken, maar vooral om zelf technologie bouwen, begrijpen en beheren. Hoe meer je zelf doet, hoe meer controle je hebt over je eigen digitale omgeving.
Er werd verteld dat we tegenwoordig erg afhankelijk zijn van grote techbedrijven. Veel apparaten zijn lastig te repareren door embedded systemen en geplande veroudering. Daarom is het belangrijk om meer controle terug te nemen door bijvoorbeeld zelf een server of website te hosten.
Ook ging het over duurzame websites met static site generators en het belang van accessibility.

Belangrijke inzichten
Zelf controle houden
Door zelf servers en websites te beheren ben je minder afhankelijk van grote bedrijven.
Static sites
Static site generators zorgen voor snelle en lichte websites met minder energieverbruik.
Duurzame websites
Afbeeldingen comprimeren, lazy loading gebruiken en system fonts kiezen maakt websites efficiënter.
Accessibility
Met prefers-reduced-motion kan je animaties uitschakelen voor gebruikers die gevoelig zijn voor beweging.
Self-hosting
Met tools zoals Python of Node.js kan je eenvoudig zelf een lokale server draaien.
Hergebruik van hardware
Oude computers kunnen nog goed gebruikt worden als server in plaats van weggegooid te worden.

Wat ik meeneem
Ik neem vooral mee dat het belangrijk is om technologie beter te begrijpen in plaats van alleen te gebruiken.
Daarnaast wil ik meer experimenteren met static sites, self-hosting en duurzame webontwikkeling zodat ik minder afhankelijk word van externe platformen.


Weekly Nerd: Johan van Stichting Accessibility
Wat hij vertelde
Tijdens deze Weekly Nerd vertelde Johan over web accessibility en hoe mensen met beperkingen websites gebruiken. Hij liet een screenreader demonstratie zien op de website van Bol.com. Het ging erg snel waardoor ik niet alles kon volgen, maar je hoorde goed hoeveel links en onderdelen een screenreader opleest.
Hij vertelde ook dat je toegankelijkheid niet goed kunt ontwerpen zonder feedback van de mensen die deze hulpmiddelen echt gebruiken.
Daarnaast besprak hij de vier basisprincipes van de WCAG:
Perceivable
Operable
Understandable
Robust
Hij legde uit dat WCAG een goede basis is, maar dat je de richtlijnen ook verkeerd kunt interpreteren.

Belangrijke inzichten
Screenreaders zijn uitgebreid
Screenreaders hebben al veel ingebouwde functies. Je hoeft daarom niet overal extra skiplinks of hulpmiddelen toe te voegen.
Gebruik echte gebruikers
Toegankelijkheid testen werkt het beste samen met mensen die daadwerkelijk beperkingen hebben.
Decoratieve afbeeldingen
Decoratieve afbeeldingen hoeven geen beschrijvende alt-tekst te hebben. Een lege alt is dan voldoende.
Goede heading structuur
Gebruikers van screenreaders navigeren vaak met headings door een pagina. Daarom is een logische heading structuur belangrijk.
Focus indicators
Duidelijke focusranden helpen gebruikers die met een toetsenbord navigeren.
Rustige interfaces
Te veel animaties, kleuren of advertenties kunnen websites onbruikbaar maken voor sommige gebruikers.
Sticky headers en animaties
Sticky headers en bewegende elementen kunnen vervelend zijn voor mensen die ingezoomd werken.
Efficiënte navigatie
Gebruikers moeten informatie snel kunnen vinden zonder te veel stappen of klikken.

Wat ik meeneem
Ik neem vooral mee dat toegankelijkheid veel verder gaat dan alleen WCAG regels volgen. Het belangrijkste is begrijpen hoe echte gebruikers websites ervaren.
Daarnaast wil ik meer testen met screenreaders en beter nadenken over rust, duidelijkheid en navigatie in mijn designs.
