Killian Valkhof

Stop using JS.. In the future

Wat hij vertelde

Tijdens de Weekly Nerd vertelde Killian Valkhof dat developers nog te veel JavaScript gebruiken terwijl HTML en CSS tegenwoordig veel mogelijkheden hebben. Zijn belangrijkste boodschap was:

kies altijd de minst krachtige oplossing die nodig is

Browsers kunnen veel sneller omgaan met HTML en CSS dan met JavaScript. Zodra je JavaScript gebruikt, moet de browser extra werk doen om alles uit te voeren en te renderen. Hij vertelde ook dat oude websites vaak nog steeds blijven werken in moderne browsers. Dat komt doordat browsers achterwaarts compatibel zijn gemaakt. Oude code blijft dus ondersteund zodat websites niet ineens kapot gaan wanneer een browser update krijgt. Als laatst liet hij voorbeelden zien van functionaliteiten die vroeger alleen met JavaScript mogelijk waren, maar nu gewoon met CSS en HTML gemaakt kunnen worden.

Belangrijke inzichten

  • Custom switch zonder JavaScript

    Met appearance: none kan je de standaard styling van een input verwijderen. Hierdoor verandert een checkbox eigenlijk in een leeg element dat je volledig zelf kunt stylen. Door vervolgens pseudo-elementen zoals ::before te gebruiken kan je bijvoorbeeld een bolletje maken binnen de switch. Met :checked kan je de staat aanpassen zodra iemand erop klikt. Hierdoor kan je een interactieve switch maken zonder één regel JavaScript te gebruiken. Killian vertelde hierbij ook dat veel developers nog denken dat forms moeilijk te stijlen zijn. Volgens hem was dat vroeger misschien waar, maar tegenwoordig bieden browsers veel betere mogelijkheden om formulier-elementen aan te passen.

  • Smooth scrolling zonder JavaScript

    Voor smooth scrolling gebruikte Killian alleen CSS: html {scroll-behavior: smooth;} Hierdoor scrollt de browser automatisch met een animatie naar een sectie van de pagina. Hiervoor hoef je dus geen JavaScript meer te gebruiken.

  • Reduce motion

    Als iemand “reduce motion” heeft aangezet in zijn browser of besturingssysteem, worden animaties uitgeschakeld. Dit is belangrijk voor toegankelijkheid en gebruiksvriendelijkheid. @media (prefers-reduced-motion: no-preference) {html {scroll-behavior: smooth;}}

  • Scroll-margin

    Met scroll-margin kan je voorkomen dat content onder een sticky header verdwijnt wanneer je naar een ankerlink scrollt.

  • Accordions met details en summary

    Killian liet zien dat je tegenwoordig accordions kunt maken met alleen HTML, zonder JavaScript te gebruiken. Met de elementen details en summary kan je eenvoudig een uitklapbaar onderdeel maken waarbij alleen de titel zichtbaar is totdat iemand erop klikt. Wanneer het onderdeel geopend wordt, voegt de browser automatisch een open status toe. Daarnaast kan je het standaard pijltje aanpassen met CSS. Ook liet hij zien dat je met het name attribuut meerdere accordions aan elkaar kunt koppelen, zodat er maar één tegelijk open kan staan. Open je een nieuwe accordion, dan sluit de vorige automatisch.

  • Dialog

    Tot slot vertelde hij over het dialog element. Dit is eigenlijk een ingebouwde popup van HTML. Het element is standaard verborgen totdat je het opent. Met :backdrop kan je de achtergrond achter de popup stijlen, bijvoorbeeld donker maken. Ook @starting-style kan gebruikt worden om dialogs te animeren.

Wat neem ik mee

Na deze Weekly Nerd wil ik bewuster omgaan met het gebruik van JavaScript. Ik wil eerst kijken of iets met HTML en CSS opgelost kan worden voordat ik direct JavaScript toevoeg. Daarnaast wil ik meer aandacht besteden aan accessibility, bijvoorbeeld door betere focus states toe te voegen en rekening te houden met gebruikers die minder animaties willen zien.

Peter Paul Koch

Browser history

Wat hij vertelde

Tijdens deze Weekly Nerd vertelde Peter-Paul Koch over de geschiedenis van browsers en hoe browsers websites renderen. Hij legde uit welke onderdelen een browser heeft, zoals HTML, CSS, de DOM en de rendering engine. Ook vertelde hij over render blocking. CSS en JavaScript kunnen ervoor zorgen dat een pagina later geladen wordt. Daarom zijn technieken zoals async en defer belangrijk voor performance. Daarnaast ging hij in op backward compatibility. Browsers blijven oude code ondersteunen zodat oude websites nog steeds blijven werken.

Belangrijke inzichten

  • Rendering engine

    Verantwoordelijk voor het verwerken van HTML en CSS en het renderen van de pagina. Browsers zoals Chrome gebruiken hiervoor Blink.

  • AOM (Accessibility Object Model)

    Naast de DOM bouwt de browser ook een AOM op. Deze wordt gebruikt door screenreaders en andere accessibility tools. Core vs DOM

  • JavaScript Core

    JavaScript Core gaat over de programmeertaal zelf, zoals variabelen en functies. De DOM zorgt ervoor dat JavaScript HTML-elementen kan aanpassen.

  • Render Blocking

    CSS blokkeert standaard rendering. JavaScript ook, tenzij je async of defer gebruikt. defer: uitvoeren nadat de DOM geladen is.

  • Backward compatibility

    Browsers blijven oude technieken ondersteunen zodat oude websites nog steeds blijven werken. Zoals het gebruiken van body bgcolor=“abdaca”>/body>

Wat neem ik mee

Ik neem vooral mee dat browsers veel complexer zijn dan ik eerst dacht. Er gebeurt veel meer achter de schermen voordat een website zichtbaar wordt.

Nils Binder

9elements

Wat hij vertelde

Tijdens deze Weekly Nerd vertelde Nils Binder over zijn bedrijf 9elements en liet hij verschillende projecten zien. Hij vertelde hoe design en development binnen hun team samenwerken om creatieve websites te bouwen. Wat ik interessant vond was dat een Figma design vaak nog erg verandert tijdens development. Het eindproduct ziet er dus niet altijd precies hetzelfde uit als het eerste ontwerp. Ook vertelde hij dat het web tegenwoordig meer mogelijkheden heeft dan Figma. Vroeger konden designs uit Photoshop vaak niet volledig op het web gemaakt worden, maar nu is dat juist andersom. Daarnaast liet hij moderne CSS technieken zien zoals subgrid, view transitions en steps animations.

Belangrijke inzichten

  • Figma lijkt op CSS

    Figma lijkt op CSS, omdat het principes gebruikt die overeenkomen met hoe je websites bouwt. Zoals auto layout die op een manier werkt die lijkt op flexbox in CSS. Met het web kan je tegenwoordig meer dan in design tools zoals Figma. Niet alles wat met CSS mogelijk is, kan ook direct in Figma worden gemaakt. Vroeger was dat andersom, toen konden designs vaak niet volledig op het web worden gerealiseerd.

  • Steps animations

    Met steps() kan je een animatie maken met meerdere afbeeldingen zodat het lijkt op een flipbook of 3D animatie/video.

  • Subgrid

    Met subgrid kunnen child elementen dezelfde grid structuur gebruiken als de parent grid.

  • View Transitions

    Met view transitions kan de browser automatisch animaties maken tussen pagina’s zonder JavaScript frameworks.

  • ::first-line

    Met ::first-line kan je alleen de eerste regel van een tekst apart stylen.

Wat neem ik mee

Ik neem vooral mee dat webdesign veel verder is dan alleen Figma namaken. CSS biedt tegenwoordig veel creatieve mogelijkheden die ik nog niet kende. Daarnaast wil ik meer experimenteren met moderne CSS features zoals subgrid en view transitions om creatievere websites te bouwen.

Robert Broersma

FF formulieren maken

Wat hij vertelde

Tijdens deze Weekly Nerd vertelde Robert Broersma over het ontwerpen van goede formulieren. Hij legde uit dat formulieren niet alleen bestaan uit invoervelden, maar dat gebruiksvriendelijkheid en toegankelijkheid een grote rol spelen. Hij vertelde bijvoorbeeld dat je alleen informatie moet vragen die echt nodig is. Ook is het belangrijk om autocomplete te gebruiken, duidelijke foutmeldingen te tonen en invoervelden logisch te ordenen. Daarnaast ging hij in op accessibility, zoals voldoende kleurcontrast, duidelijke labels en grote klikvlakken voor gebruikers.

Belangrijke inzichten

  • Vraag alleen wat nodig is

    Vraag alleen gegevens die echt noodzakelijk zijn. Volgens de wet mag je geen onnodige informatie verzamelen. Geef daarnaast duidelijk aan welke velden optioneel zijn, in plaats van alle verplichte velden te markeren.

  • Maak Invullen Makkelijk

    Gebruik autocomplete en zorg voor grote klikvlakken van ongeveer 44x44px. Dit maakt formulieren sneller, toegankelijker en prettiger om in te vullen.

  • Zorg Voor Duidelijke Structuur

    Elk invoerveld moet een zichtbaar label hebben en de velden moeten in een logische volgorde staan. Daardoor voelt het formulier natuurlijker aan en begrijpen gebruikers sneller wat ze moeten doen.

  • Help Gebruikers Bij Fouten En Onderbrekingen

    Na het verzenden/of direct bij het invullen moet duidelijk zichtbaar zijn welke velden fouten bevatten. Bij langere formulieren is het handig als gebruikers hun voortgang kunnen opslaan en later verder kunnen gaan.

Wat neem ik mee

Ik neem vooral mee dat kleine details een formulier veel gebruiksvriendelijker maken. Dingen zoals autocomplete, duidelijke labels en goede foutmeldingen maken een groot verschil voor de gebruiker. Daarnaast wil ik bij toekomstige formulieren meer rekening houden met toegankelijkheid en duidelijkheid.

Hacker Filosofie

Wat hij vertelde

Tijdens deze Weekly Nerd ging het over hacker filosofie. Hierbij draait het niet alleen om hacken, maar vooral om zelf technologie bouwen, begrijpen en beheren. Hoe meer je zelf doet, hoe meer controle je hebt over je eigen digitale omgeving. Er werd verteld dat we tegenwoordig erg afhankelijk zijn van grote techbedrijven. Veel apparaten zijn lastig te repareren door embedded systemen en geplande veroudering. Daarom is het belangrijk om meer controle terug te nemen door bijvoorbeeld zelf een server of website te hosten.

Belangrijke inzichten

  • Zelf Controle Over Je Website

    Door zelf servers en websites te beheren ben je minder afhankelijk van grote bedrijven. Met self-hosting en tools zoals Python of Node.js kan je eenvoudig zelf een lokale server draaien.

  • Snelle En Lichte Websites

    Static site generators maken websites sneller en lichter. Omdat er minder serverwerk nodig is, gebruiken deze websites vaak ook minder energie.

  • Duurzamer Webdesign

    Je kunt websites efficiënter maken door afbeeldingen te comprimeren, lazy loading te gebruiken en system fonts te kiezen. Daardoor laadt de website sneller en verbruikt hij minder data en energie.

  • Hergebruik En Toegankelijkheid

    Oude computers kunnen nog goed gebruikt worden als server in plaats van weggegooid te worden. Daarnaast kan je met prefers-reduced-motion animaties uitschakelen voor gebruikers die gevoelig zijn voor

Wat neem ik mee

Ik neem vooral mee dat het belangrijk is om technologie beter te begrijpen in plaats van alleen te gebruiken en dat je oude tech nog prima kan gebruiken voor onderdelen of anderen doeleinde.

Johan

Stichting Accessibility

Wat hij vertelde

Tijdens deze Weekly Nerd vertelde Johan over web accessibility en hoe mensen met beperkingen websites gebruiken. Hij liet een screenreader demonstratie zien op de website van Bol.com. Het ging erg snel waardoor ik niet alles kon volgen, maar je hoorde goed hoeveel links en onderdelen een screenreader opleest. Hij vertelde ook dat je toegankelijkheid niet goed kunt ontwerpen zonder feedback van de mensen die deze hulpmiddelen echt gebruiken. Daarnaast besprak hij de vier basisprincipes van de WCAG:
- Perceivable: Informatie moet waarneembaar zijn, bijvoorbeeld door tekstalternatieven voor afbeeldingen.
- Operable: Gebruikers moeten de interface kunnen bedienen, bijvoorbeeld met een toetsenbord.
- Understandable: Informatie en bediening moeten begrijpelijk zijn, bijvoorbeeld door duidelijke taal en consistente navigatie.
- Robust: Inhoud moet robuust genoeg zijn om betrouwbaar te werken.

WCAG is een goede basis, maar kan je ook verkeerd kunt interpreteren.

Belangrijke inzichten

  • Bouw Op Bestaande Hulpmiddelen

    Screenreaders hebben al veel ingebouwde functies. Daarom hoef je niet overal extra skiplinks of hulpmiddelen toe te voegen. Decoratieve afbeeldingen hoeven ook geen beschrijvende alt-tekst te hebben. Een lege alt is dan voldoende.

  • Test Met Echte Gebruikers

    Toegankelijkheid testen werkt het beste samen met mensen die daadwerkelijk beperkingen hebben. Zij ervaren problemen die je zelf vaak niet ziet tijdens het bouwen.

  • Zorg Voor Duidelijke Navigatie En Structuur

    Gebruikers van screenreaders navigeren vaak met headings door een pagina. Daarom is een logische heading structuur belangrijk. Ook moeten gebruikers informatie snel kunnen vinden zonder te veel stappen of klikken.

  • Maak De Interface Rustig En Bedienbaar

    Duidelijke focus state helpen gebruikers die met een toetsenbord navigeren. Vermijd daarnaast te veel animaties, kleuren, advertenties, sticky headers en bewegende elementen, omdat die websites onbruikbaar of vervelend kunnen maken voor sommige gebruikers.

Wat neem ik mee

Ik neem vooral mee dat toegankelijkheid veel verder gaat dan alleen WCAG regels volgen. Het belangrijkste is begrijpen hoe echte gebruikers websites ervaren. Daarnaast wil ik meer testen met screenreaders en beter nadenken over rust, duidelijkheid en navigatie in mijn designs.
